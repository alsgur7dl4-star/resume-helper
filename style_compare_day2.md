# Day 2 스타일 비교 로그

## 입력 문단

SI 자바 웹개발 3개월 인턴을 마쳤습니다. SI/SM 프로젝트를 진행하면서 배치 실행 중 발생한 오류를 처리하는 방법을 경험했고, 여러 기능을 하나로 합치는 역할을 맡았습니다.

## 기본 스타일 3종

### 간결형

개선 필요(우선순위)

1. **무엇을/어떻게 해결했는지 구체화**: “오류 처리 방법”이 비어 있어요. 예) 원인 분석 절차, 로그/모니터링, 재발 방지 조치(알림, 예외처리 규칙, 배치 재시도 등).
2. **정량 성과 추가**: 오류 대응 건수/감소율/장애 시간 단축/성공률 같은 숫자가 없어요. (예: 월 장애 3회→1회, 배치 성공률 98% 등)
3. **문장 구조 다듬기(두괄식)**: 현재는 정보가 나열돼 읽히기 어렵습니다. “나는 O를 했고, 그 결과 P를 만들었다” 형태로 줄이세요.
4. **용어 일관성**: “si/sm”이 무엇인지(단위/범위) 자소서 문맥에서 한 번만 정리하면 좋아요.

### 스토리형

### (1순위) 경험의 ‘구체 장면’이 부족함

- “배치가 돌아가면서 생긴 오류”라고 했지만,
  - 어떤 배치(예: 야간 정산, 데이터 적재, 배치 스케줄러)?
  - 어떤 유형의 오류(예: 데이터 정합성, 타임아웃, 중복 적재, 권한/스키마 오류)?
  - 오류가 **어떻게 발견/재현**됐고, **어떻게 해결**했는지  
    가 비어 있어요. 그래서 읽는 사람이 “그래서 어떤 방식으로 성장했는지”까지 못 따라갑니다.

### 직무맞춤형

- 지금 문장은 **직무 역량에 ‘어떤 방식으로, 어떤 결과를’ 냈는지(정량/구체성)**가 부족해서 설득력이 약해요.
- SI 자바 웹개발 직무 관점에서 보완할 점을 우선순위로 정리해드릴게요.

## 1순위(가장 중요): “오류 처리”를 방법+성과로 구체화해야 함

### 현재 문제

- 어떤 오류인지(원인/유형: 타이밍, 파라미터, 장애 전파, 스케줄 충돌 등)
- 어떻게 처리했는지(분석 절차, 로그/모니터링, 재처리 로직, 예외처리 설계 등)

### 친근한 버전

- SI 자바 웹개발 인턴을 3개월간 마치며, SI/SM 프로젝트에서 배치 작업이 돌아갈 때마다 발생하는 오류를 원인별로 정리하고 재발을 줄이는 방식으로 대응했습니다. 또한 여러 기능을 하나로 묶어 동작 흐름을 맞추는 역할을 맡아, 서로 다른 컴포넌트가 안정적으로 연동되도록 점검·수정했습니다.

### AI 서비스 연결형

- SI/SM 프로젝트에서 배치 실행 중 발생한 오류를 확인하고 처리하는 경험을 했습니다. 또한 여러 기능을 하나로 합치는 역할을 맡으며 기능 간 흐름과 연동 구조를 이해했습니다. 이 경험은 이후 AI 서비스를 개발할 때 오류 대응, 데이터 흐름 관리, 기능 통합 역량으로 연결될 수 있습니다.
